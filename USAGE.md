# USAGE.md — rehydrate end-to-end walkthrough

## Running a backup

**Prerequisites:** A mounted external drive with at least 500 MB free. The drive path must
begin with `/Volumes/` (e.g. `/Volumes/PortableSSD`).

**Invoke the backup skill in Claude Code:**

```
/rehydrate:backup
```

The skill will prompt you for:

1. **Drive path** — absolute path to the mounted backup drive.
2. **Snapshot label** (optional) — a short alphanumeric label appended to the snapshot ID,
   e.g. `pre-upgrade`. Useful for distinguishing snapshots taken at notable moments.
3. **Parent chaining** — if prior snapshots exist on the drive, the skill will offer to chain
   the new snapshot as a child of the most recent one (recommended). Parent chaining records
   lineage in `parent.txt` and enables diff and GC operations.

**What the skill produces on the drive:**

```
<drive>/llm-backup/
  objects/
    <aa>/
      <bb>/
        <sha256full>          # content-addressed, deduped across all snapshots
  snapshots/
    <hostname>-<timestamp>[-label>/
      manifest.json           # machine-readable inventory (files, hashes, categories)
      RESTORE-GUIDE.md        # narrative guide generated by the backup LLM
      parent.txt              # parent snapshot ID, or "none"
      probe.json              # source machine metadata
      walk-<category>.json    # per-category walker output
      workdirs/               # staging dirs for package-list/app-list/repo-list walkers
```

The skill reports per-category file counts, any skipped files, and total drive usage after
a successful backup.

If there are more than 10 snapshots on the drive, the skill will suggest running GC (see
"Pruning old snapshots" below). It will not run GC automatically.

## Running a restore

**Invoke the restore skill in Claude Code:**

```
/rehydrate:restore
```

The skill collects:

1. **Snapshot path** — full path to a snapshot directory, e.g.
   `/Volumes/PortableSSD/llm-backup/snapshots/<id>`.
2. **Target path** (optional, defaults to `$HOME`) — where to write restored files. Specify
   a temp directory (e.g. from `mktemp -d`) for a safe sandbox test before any live restore.

**The dual-confirmation flow:**

- **Checkpoint 1:** The skill runs `restore-plan.py`, displays a summary of actions
  (create / skip-identical / overwrite-needs-confirm), repeats any drift warnings, and waits
  for explicit confirmation. Restoring to `$HOME` requires typing `yes, live` exactly.
- **Dry run:** Before writing anything, the skill runs `restore-apply.py --dry-run` and
  displays the output.
- **Checkpoint 2:** After the dry run, the skill asks for final confirmation before the live apply.

**Drift warnings** (from comparing snapshot source machine to current machine):

| Drift | Behaviour |
|---|---|
| Different OS platform (e.g. Linux snapshot on macOS) | Hard refusal — no override |
| Architecture mismatch (arm64 vs x86_64) | Requires typing `yes, I understand the arch mismatch` |
| Major OS version difference | Displayed as a prominent warning; user may continue |
| Minor OS or hostname difference | Informational only |

**Where each strategy's output lands after restore to `<target>`:**

| Category (strategy) | Restore location |
|---|---|
| dotfiles, ssh-keys, gnupg, launchagents, defaults, browser-profiles, app-sessions (file-list) | `<target>/` — files are placed at their original relative paths under `$HOME` |
| package-managers (package-list) | `<target>/.rehydrate/packages/` — Brewfile, pip-requirements.txt, npm-globals.json, etc. |
| app-inventory (app-list) | `<target>/.rehydrate/apps/inventory.json` |
| dev-projects (repo-list) | `<target>/.rehydrate/projects/inventory.json` + captured secrets per repo |
| local-only-projects (full-snapshot) | `<target>/.rehydrate/projects-local/<name>/` |

After restoring package inventories, re-install manually:

```bash
brew bundle install --file=<target>/.rehydrate/packages/Brewfile
pip3 install -r <target>/.rehydrate/packages/pip-requirements.txt
```

For git repos, clone each from the remote URL recorded in the inventory, then copy
captured secrets (`.env`, `*.p8`, `credentials.json`) back into each repo working tree.
After restoring the `defaults` category, run `killall cfprefsd` or log out and back in for
macOS to pick up the refreshed preference plists.

## Inspecting changes between snapshots

`snapshot-diff.py` compares two snapshots and reports files added, removed, or changed
between them. It operates on the parent chain by default, or you can specify an explicit
comparison target.

```bash
# Compare against parent snapshot (requires parent chaining):
python3 scripts/snapshot-diff.py \
  --drive /Volumes/PortableSSD/llm-backup \
  --snapshot <snapshot-id>

# Compare against a specific earlier snapshot:
python3 scripts/snapshot-diff.py \
  --drive /Volumes/PortableSSD/llm-backup \
  --snapshot <snapshot-id> \
  --against <older-snapshot-id>
```

Output lists files by status (added / removed / changed) with their category tags.

## Pruning old snapshots

`snapshot-gc.py` removes snapshots (and unreferenced objects) to reclaim drive space.
It uses reachability: objects referenced by any kept snapshot are never deleted.

Dry run first — see what would be removed without deleting anything:

```bash
python3 scripts/snapshot-gc.py \
  --drive /Volumes/PortableSSD/llm-backup \
  --keep-last 5 \
  --dry-run
```

Apply the prune:

```bash
python3 scripts/snapshot-gc.py \
  --drive /Volumes/PortableSSD/llm-backup \
  --keep-last 5
```

`--keep-last N` retains the N most recent snapshots and removes older ones. The object
store is pruned after snapshot removal: any object not referenced by a remaining snapshot
is deleted. Always run `--dry-run` first to review what will be removed.

## Verifying a snapshot

`verify-sandbox.py` performs a sandboxed test restore — it drives the full restore
procedure into a temp directory and diffs the result against the snapshot. Run it after
every backup to confirm the snapshot is intact and the restore logic is working:

```bash
python3 scripts/verify-sandbox.py \
  --snapshot /Volumes/PortableSSD/llm-backup/snapshots/<snapshot-id>
```

The script creates an isolated temp directory, restores all files there, and compares
content hashes against the manifest. On success it deletes the sandbox and reports
pass. On failure it preserves the sandbox at the printed path for inspection.

Run verification any time you want confidence that a stored snapshot is restorable,
especially before relying on a snapshot for a real migration.

## Customising categories.yaml

The `categories.yaml` file in the plugin directory controls which categories are enabled
and what globs each one captures. You can edit a local copy to adjust behaviour.

**Example 1 — Tighten browser-profiles excludes** to also skip the `History` and
`Cookies` files (reducing snapshot size and avoiding stale session data):

```yaml
- name: browser-profiles
  enabled: true
  strategy: file-list
  # ... existing globs ...
  exclude:
    - "**/Cache/**"
    - "**/Code Cache/**"
    # add:
    - "**/History"
    - "**/Cookies"
```

**Example 2 — Populate custom-content globs** to capture a local music library and
a design assets folder not covered by any other category:

```yaml
- name: custom-content
  enabled: true
  strategy: file-list
  globs:
    - ~/Music/Logic/**
    - ~/Documents/Design/Assets/**
```

**Example 3 — Disable a category for a specific machine** that has no relevant data
(e.g. a machine where no git repos live under `~/Documents/Projects`):

```yaml
- name: dev-projects
  enabled: false   # change from true to false
  strategy: repo-list
  # ... rest unchanged ...
```

Disabled categories are skipped entirely — no walk is attempted and no entries appear
in the manifest.

## Restoring on a fresh Mac

High-level procedure for a clean-Mac migration:

1. **Install rehydrate** — follow [INSTALL.md](INSTALL.md) Path B (clone + local plugin install).

2. **Mount the backup drive** — connect the external drive containing your snapshot.
   Verify it appears at the expected path (e.g. `/Volumes/PortableSSD`).

3. **Invoke restore:**
   ```
   /rehydrate:restore
   ```
   When prompted, provide the full snapshot path:
   `/Volumes/PortableSSD/llm-backup/snapshots/<snapshot-id>`

   Start with a sandbox target (a temp dir) to verify the snapshot first, then repeat
   with `$HOME` as the target (requires `yes, live` confirmation).

4. **Post-restore steps per category:**

   - **dotfiles** — open a new terminal session to pick up restored shell configs.
   - **ssh-keys** — `chmod 600 ~/.ssh/id_*` if file modes are not preserved.
   - **package-managers** — run `brew bundle install`, `pip3 install -r`, etc. from
     `<target>/.rehydrate/packages/`.
   - **app-inventory** — work through `<target>/.rehydrate/apps/inventory.json` to reinstall
     apps via `brew install --cask`, Mac App Store, or manual download.
   - **defaults** — run `killall cfprefsd && killall Finder && killall Dock` to reload
     preference plists and restart the affected system components.
   - **dev-projects** — clone each repo from its recorded remote URL, then copy the captured
     secrets back:
     ```bash
     git clone <remote_url> ~/Documents/Projects/<name>
     cp -R <target>/.rehydrate/projects/<name>/. ~/Documents/Projects/<name>/
     ```
     Review every restored `.env` / `.p8` / `credentials.json` before use — API keys may
     have been rotated since the backup was taken.
   - **local-only-projects** — `mv <target>/.rehydrate/projects-local/<name> ~/Documents/Projects/<name>`
