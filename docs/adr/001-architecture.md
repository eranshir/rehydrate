# ADR-001: rehydrate v0.1 — Core Architecture Decisions

This ADR captures nine architectural decisions made during the scoping of **rehydrate v0.1**. The goal is to give future implementers, reviewers, and the restore-side LLM a single place to understand the *why* behind each structural choice so those decisions are not re-litigated implementation by implementation.

**Thesis recap.** Traditional backups store bytes. rehydrate stores the knowledge that the bytes can be re-derived from. Apps, packages, runtimes, and repository contents shrink to a list; the restore LLM re-derives them. What stays full-size: secrets, profiles, sessions, locally-authored files, and custom configs — anything that is not already on the public internet or reproducible from a canonical source.

## 1. Hybrid manifest format

### Context

A snapshot must be self-describing. A restore performed months or years after backup — possibly by a different model version, possibly on a different OS version — must be able to reconstruct the machine state without any external documentation. The manifest is the single artifact that bridges snapshot time and restore time.

Two natural approaches exist: a structured schema (JSON, YAML) that is machine-parseable, and a narrative document (Markdown) that a language model can read as prose context. Neither alone is sufficient.

A restore LLM needs more than a file list. It needs to know *why* certain files were included, what the user's priorities were, which categories are optional vs. essential, and what platform assumptions underpin each restore step. Without that narrative, the restore LLM must either guess or ask the user repeatedly — both failures of the backup's self-describing goal.

At the same time, a narrative-only format forces the restore LLM to parse structure on every pass. File paths, hashes, and category membership must be extracted by the model rather than looked up, making the restore path fragile and slow.

### Decision

The snapshot root contains two co-equal manifest artifacts:

- **`manifest.json`** — A machine-readable inventory. Contains: snapshot metadata (UUID, timestamp, hostname, OS version), a list of all captured files with their SHA-256 content hash and category tag, the exclusion list applied at backup time, and a reference to the category definitions used.
- **`RESTORE-GUIDE.md`** — A narrative document generated by the backup skill. Contains: a human-readable summary of what was captured and why, any unusual decisions made during the backup run (e.g., files skipped due to size, categories found empty), the user's stated restore priorities if provided, and a step-by-step reasoning scaffold for the restore LLM.

The two documents are generated together during every backup run. `manifest.json` is the source of truth for what to restore; `RESTORE-GUIDE.md` is the source of truth for how to interpret edge cases.

### Consequences

- The restore skill can index `manifest.json` directly for file lookup, hash verification, and category iteration.
- The restore skill can prompt the LLM with `RESTORE-GUIDE.md` as context for judgment calls without re-parsing structure.
- Both documents must be kept in sync; the backup skill is the single writer of both.
- `RESTORE-GUIDE.md` is generated prose, not hand-edited documentation — it reflects the actual backup run, not an idealized description.
- Snapshot size increases slightly (two manifest files instead of one), but both are small relative to the object store.

### Alternatives considered

**Schema-only (JSON manifest, no narrative).** Rejected. The restore LLM would have only file lists and hashes. Judgment calls — which categories to restore first, how to handle conflicts, what to do when a file already exists on the target — require context that a schema cannot carry. The restore LLM would be forced to ask the user or guess, defeating the self-describing goal.

**Narrative-only (single Markdown document).** Rejected. Every restore operation would require the LLM to extract structure (file paths, hashes, category membership) from prose. This is fragile, slow, and error-prone at scale. A 500-file snapshot described only in prose is unacceptable.

## 2. Content-addressed `objects/` storage

### Context

A backup drive accumulates multiple snapshots over time. Without deduplication, every snapshot is a full copy of every captured file. A 2 GB snapshot taken weekly produces 100 GB of storage in a year, even if most files never change.

The backup drive is a local external drive, not a cloud service with built-in dedup. Storage efficiency must be implemented by rehydrate itself.

Files also need to be verifiable. A restore must be able to confirm that the bytes it is about to write match the bytes that were captured. An address scheme that incorporates the file's content hash provides this for free.

### Decision

Files are stored in a flat content-addressed object store at:

```
<drive>/llm-backup/objects/<aa>/<bb>/<sha256full>
```

Where `aa` is the first two hex characters of the SHA-256 hash, `bb` is the next two hex characters, and `sha256full` is the full 64-character hex hash. This is a two-level fanout, matching the scheme used by Git's loose object store.

No file is stored twice. Before writing an object, the snapshot script checks whether `objects/<aa>/<bb>/<hash>` already exists. If it does, the object is referenced in the manifest but not rewritten.

`manifest.json` references each captured file by its content hash. Restore assembles files by looking up objects in the store by hash. The object store is append-only; objects are never modified or deleted during normal operation (pruning is a Phase 4 concern).

### Consequences

- Cross-snapshot deduplication is automatic and exact: identical file content at different paths or across different snapshots is stored once.
- File integrity verification is free: hash the object, compare to the hash in the manifest.
- The object store is inspectable with standard filesystem tools — no proprietary database.
- Objects are immutable after write, which simplifies the restore path and eliminates write-after-read races.
- Object filenames are not human-readable; navigating the object store without the manifest is not intended to be ergonomic.
- Pruning (removing objects not referenced by any live snapshot) is deferred to Phase 4. In v0.1 the store only grows.

### Alternatives considered

**Per-snapshot copies (one directory per snapshot, full file copies).** Rejected. Storage usage grows linearly with snapshot count multiplied by total captured size. A weekly backup habit makes this untenable within months. No mechanism for cross-snapshot dedup exists without content addressing.

**rsync-style binary deltas.** Rejected. Delta storage is more space-efficient than full copies for slowly-changing large files, but it adds complexity: a delta chain must be traversed to reconstruct any given version, recovery from a broken chain is difficult, and the store cannot be inspected without tooling. The space savings do not justify the complexity for the file sizes typical in this backup category (dotfiles, configs, keys). Content addressing with full-object storage is simpler, more robust, and adequate given that most captured files are small.

## 3. Sandboxed test restore as the verify strategy

### Context

A backup that cannot be verified is not trustworthy. Verification must confirm not only that the manifest is well-formed, but that the actual restore procedure — reading objects, reconstructing paths, applying files — produces the expected result.

Verification must be runnable frequently (ideally after every backup) and must not risk corrupting the live machine state. It should catch both manifest bugs (wrong path, wrong hash) and restore-logic bugs (wrong file mode, wrong directory creation order).

### Decision

Verification is implemented as a sandboxed test restore. The verify script (`scripts/verify-sandbox.py`) creates a temporary directory tree isolated from `$HOME`, then drives the full restore procedure against that target. After restore completes, it diffs the reconstructed tree against the snapshot's object store to confirm that every captured file was reconstructed exactly.

The key invariant: `$REHYDRATE_TARGET` is always set to the sandbox path during verification. The restore scripts check this variable and refuse to write outside it. Live `$HOME` is never touched during a verification run.

Verification is not a separate mode — it is the same restore code path, with a different target. This means verification exercises the actual restore logic, not a separate dry-run simulation.

After verification passes, the sandbox is deleted. If verification fails, the sandbox is preserved for inspection.

### Consequences

- Verification catches both manifest bugs and restore-logic bugs.
- No risk of corrupting live state: `$REHYDRATE_TARGET` is enforced as a safety boundary.
- Verification uses the same code path as live restore, so passing verification gives high confidence that live restore will also succeed.
- Verification requires disk space for the sandbox (up to one full snapshot size). This is acceptable on external drives but should be documented.
- Verification time scales with snapshot size and the number of files. For large snapshots this may take several minutes.
- The verification diff catches missing files and content mismatches but does not verify file modes or extended attributes in v0.1. These may be added in later phases.

### Alternatives considered

**Dry-run only (parse manifest, report what would be restored, do not write files).** Rejected. A dry-run catches manifest parsing bugs and path resolution bugs, but it does not exercise the object store lookup, the file write logic, or the directory creation order. A backup that passes dry-run and then fails live restore provides false confidence, which is worse than no verification.

**VM-based verification (Tart or UTM, full macOS VM, restore into a clean OS).** Not rejected permanently — this is the most rigorous possible verify strategy. But it is too heavy for routine use: VM provisioning takes 30+ minutes per cycle, requires a macOS installer image, and consumes significant CPU and disk resources. Sandboxed temp-dir verify catches the majority of bugs with a fraction of the overhead. VM-based verification remains a valid future enhancement (Phase 5 or later) for pre-release confidence testing.

## 4. iCloud-trusted exclusion list

### Context

macOS keeps the canonical copy of many user data categories in iCloud. Mail, Notes, Photos, Safari browsing data, Contacts, Calendar, Reminders, Voice Memos, Messages, and App Store purchase history are all synced and recoverable from iCloud without a local backup. Capturing them locally re-invents what iCloud already does, wastes backup storage, and in some cases (Keychain, Messages) creates legal or security sensitivity.

The backup must have an opinionated default that a normal user need not override. A backup that requires the user to manually enumerate everything to exclude before it is safe to run is a bad default.

### Decision

The following categories are excluded from backup by default:

| Category | iCloud/System source |
|---|---|
| Mail (`~/Library/Mail`) | iCloud Mail |
| Notes (`~/Library/Group Containers/group.com.apple.notes`) | iCloud Notes |
| Photos (`~/Pictures/Photos Library.photoslibrary`) | iCloud Photos |
| Safari (`~/Library/Safari`) | iCloud Safari |
| Keychain (`~/Library/Keychains`) | iCloud Keychain |
| Contacts (`~/Library/Application Support/AddressBook`) | iCloud Contacts |
| Calendar (`~/Library/Calendars`) | iCloud Calendar |
| Reminders (`~/Library/Reminders`) | iCloud Reminders |
| Voice Memos (`~/Library/Application Support/com.apple.voicememos`) | iCloud Voice Memos |
| Messages (`~/Library/Messages`) | iCloud Messages |
| App Store (`~/Library/Application Support/AppStore`) | App Store purchase history |

These paths are listed in `categories.yaml` under the `icloud-trusted-exclusions` section. The backup skill logs which paths it skipped due to this list.

Users can override individual exclusions by removing them from their local `categories.yaml` override file, but the default ships with all entries active.

### Consequences

- Default backups are smaller and faster because they skip large iCloud-managed directories.
- The backup does not create a redundant copy of data already protected by iCloud.
- Users with iCloud disabled for specific categories must remember to override the exclusion list, or their data will not be backed up. This is a known tradeoff documented in the README.
- The exclusion list encodes a Mac-specific trust assumption (iCloud is the right canonical source). This is acceptable for v0.1 which targets macOS only.
- Future Linux/Windows support would require a different exclusion list and a different trust model.

### Alternatives considered

**Capture everything (no exclusions).** Rejected. This re-invents iCloud for data that iCloud already manages reliably. It doubles storage usage for the most common user data categories. It creates copies of Keychain data on an external drive in plaintext (see Decision 5), which is a security concern even for physically trusted drives.

**No exclusions by default (user must configure).** Rejected. A backup system with no safe defaults shifts a difficult configuration burden onto the user. Most users will not configure exclusions correctly before running their first backup, leading to oversized and potentially insecure snapshots. A bad default that requires expertise to fix is a product failure.

## 5. Plaintext secrets on backup drive in v1

### Context

rehydrate captures secrets: SSH private keys, `.env` files with API tokens, GPG private keyrings, and similar material. These must be captured, because they cannot be re-derived from any public source. Without them, a restore is incomplete.

Encryption at rest would protect these secrets if the backup drive is lost or stolen. However, encryption introduces key management complexity: the key must be available at restore time, possibly years later, possibly on a machine where the original key storage is unavailable. Getting key management wrong can make backups permanently unrecoverable.

v0.1 is a working prototype. The primary goal is correctness and completeness; a sophisticated key management scheme in the first release risks correctness bugs and complicates the restore path.

### Decision

Secrets are stored **plaintext** on the backup drive in v0.1. The drive is assumed to be physically trusted (encrypted at the drive level via FileVault, or a trusted external drive in the user's physical possession).

Every backup run prints a visible security notice reminding the user of this assumption. The README prominently documents it. The notice reads:

> **Security notice:** Secrets (SSH keys, .env files, API tokens) are stored plaintext on the backup drive. Ensure the drive is physically secure or encrypted at the device level (e.g., FileVault on an external drive). Encrypted-at-rest secrets are planned for v0.2.

The plaintext-on-drive approach is explicitly scoped to v0.1. The v2 upgrade path is documented below.

**v2 upgrade path.** Two encryption approaches are planned for evaluation in a later phase:

1. **age encryption with a skill-managed key.** The backup skill generates an [age](https://github.com/FiloSottile/age) identity at setup time and stores the public key in `manifest.json`. Each secret object is encrypted to that public key. The private key is stored in the user's Keychain (or another trusted location chosen at setup). Restore requires access to the private key. This approach has excellent tooling, is scriptable, and is auditable.

2. **Vault-backed encryption (1Password CLI / aegis-secret).** Secret objects are not stored on the backup drive at all. Instead, the manifest records the 1Password item reference (or aegis-secret key name). Restore retrieves each secret from the vault at restore time. This is the most secure approach but requires the vault to be accessible at restore time — a hard dependency.

The v2 decision between these approaches is deferred. The v0.1 object store format is designed so that object contents can be replaced with encrypted versions without changing the manifest schema or the restore logic (the object lookup path is the same; the restore script just needs to decrypt after lookup).

### Consequences

- v0.1 ships faster without key management complexity.
- The restore path is simpler: read object, write file. No decryption step.
- The backup drive must be physically trusted. This is a real security assumption that must be clearly communicated.
- If the drive is lost without device-level encryption, all captured secrets are exposed.
- Upgrading to encrypted secrets in v2 requires re-encrypting existing object stores. A migration script will be needed.

### Alternatives considered

**age/gpg encryption in v0.1.** Rejected for v0.1 due to key management complexity. If the age identity is lost, the backup is permanently unrecoverable. Getting key generation, storage, and restore-time retrieval right on the first version adds significant risk without a clear correctness-testing strategy. This is the right approach for v2 once the rest of the system is stable.

**Vault-backed (1Password CLI / aegis-secret) in v0.1.** Rejected for v0.1. Requires the 1Password CLI or aegis-secret to be installed and authenticated on both the source and target machines. This adds a mandatory dependency and a hard restore-time requirement (vault access) that may not be satisfiable years after backup. Suitable for v2 as an opt-in for users who already use these tools.

## 6. No-PII logging principle

### Context

Helper scripts run during backup and restore log progress, errors, and diagnostic information. Without a logging discipline, it is easy for a debug print statement to inadvertently log file contents — and file contents may include secrets, personal documents, or other sensitive material.

Logs may be:
- Displayed in the terminal (visible to bystanders)
- Sent to Claude Code's conversation context (potentially shared with Anthropic)
- Written to log files (potentially included in bug reports)

A logging approach that relies on programmer discipline ("don't log contents") is fragile. Any script author adding a debug line must remember the constraint. Over time, as the codebase grows, violations are likely.

### Decision

All logging in rehydrate helper scripts goes through a shared module: `scripts/no_pii_log.py`.

This module:
- Exposes functions for logging path names, SHA-256 hashes, counts, error codes, and status strings.
- **Does not expose any function that accepts arbitrary string content.** There is no `log_content()`, no `log_bytes()`, no generic `log_debug(msg)` that could be called with file contents.
- Is the only sanctioned logging interface. Scripts that need to log anything import this module; they do not write directly to stdout/stderr for diagnostic output.

The module's API surface is the enforcement mechanism. A developer cannot accidentally log file contents through this module because the module has no API for it. Bypassing the module (writing to stdout directly) is visible in code review and grep-able (`grep -r 'print(' scripts/`).

Example sanctioned log calls:

```python
from no_pii_log import log_file_captured, log_hash_skipped, log_error

log_file_captured(path="~/.zshrc", sha256="a3f9...")
log_hash_skipped(sha256="a3f9...", reason="already in object store")
log_error(script="snapshot.py", code="EACCES", path="~/Library/Keychains")
```

Example unsanctioned pattern (caught in review):

```python
# BAD — logs file content
print(f"DEBUG: file contents = {open(path).read()}")
```

### Consequences

- Accidental content logging requires a deliberate bypass of the logging module, making it visible and reviewable.
- All log output has a known, auditable format.
- New script authors must learn the `no_pii_log` API, but the module's narrow surface makes this quick.
- The module must be kept in sync with the full set of useful log events; if a new log type is needed, the module must be extended rather than bypassed.
- Scripts cannot use arbitrary `print()` calls for diagnostics without flagging in review.

### Alternatives considered

**Trust scripts not to log contents (no shared module, convention only).** Rejected. Convention-based discipline erodes over time. Debug prints are the most common source of accidental data leaks in scripts. A shared module with a narrow API is a structural enforcement mechanism that does not rely on per-developer vigilance. The cost of the module is low; the cost of a leaked secret in a conversation log is high.

## 7. Plugin (not standalone skill) packaging

### Context

rehydrate provides two user-facing operations: backup and restore. These must be exposed as slash commands in Claude Code. The Claude Code extension model offers two relevant packaging options: a **skill** (one slash command) and a **plugin** (a bundle that can contribute multiple slash commands and helper scripts).

Current Claude Code convention is that one skill maps to one slash command. A skill that accepts an argument to route between backup and restore behavior would violate this convention and produce a confusing user experience (`/rehydrate backup` vs. `/rehydrate:backup`).

### Decision

rehydrate is packaged as a **Claude Code plugin** defined by `.claude-plugin/plugin.json`. The plugin contributes:

- `/rehydrate:backup` — the backup skill
- `/rehydrate:restore` — the restore skill
- Helper scripts in `scripts/` (bundled with the plugin, not installed globally)
- Category definitions in `categories.yaml`
- Schema definitions in `schemas/`

The two skills are independent entry points. They share helper scripts and the manifest schema, but each has its own `SKILL.md` frontmatter and its own prompt logic.

### Consequences

- Users invoke clearly separated commands (`/rehydrate:backup` vs. `/rehydrate:restore`) with no ambiguity about which operation is being triggered.
- Each skill can evolve its frontmatter (model, context window, disable-model-invocation) independently.
- Plugin packaging is the correct extension point for multi-command tools in Claude Code.
- The plugin must be installed as a unit; there is no way to install backup without restore or vice versa. This is acceptable — the two operations are complementary.
- Helper scripts are bundled with the plugin and do not need to be on the user's `$PATH`.

### Alternatives considered

**One skill with argument routing (`/rehydrate backup` or `/rehydrate restore`).** Rejected. This violates the current Claude Code convention that one skill maps to one slash command. Argument-routed skills produce a command-line feel rather than a slash-command feel. It also makes it harder to set different frontmatter options (e.g., model, disable-model-invocation) for backup vs. restore. The routing logic adds unnecessary complexity to the skill entry point.

## 8. `disable-model-invocation: true` on skill frontmatter

### Context

Claude Code skills can be configured to allow the model to auto-invoke them as part of agentic reasoning — if the model decides a backup would be useful in the middle of another task, it could trigger `/rehydrate:backup` autonomously.

Backup and restore are **side-effecting, drive-writing operations**. A backup triggered at an unexpected moment could:
- Write several gigabytes to an external drive without user awareness
- Interfere with a drive that is in use
- Cause snapshot ID collisions if triggered during an existing backup
- Trigger during situations where the drive is not mounted

A restore triggered autonomously could overwrite files on the target without explicit user confirmation.

### Decision

Both skills (`backup` and `restore`) set `disable-model-invocation: true` in their `SKILL.md` frontmatter. The model cannot trigger either skill as a tool call. The user must type `/rehydrate:backup` or `/rehydrate:restore` explicitly.

This is the maximum restriction available in the Claude Code skill frontmatter. It means rehydrate operations are always deliberate, user-initiated actions.

### Consequences

- No risk of an autonomous backup or restore being triggered during unrelated agentic work.
- Users must always explicitly invoke backup or restore. This is the correct UX for operations that write to an external drive.
- The model cannot suggest and then immediately execute a backup — it can only suggest that the user run the command themselves.
- If future versions of rehydrate add a scheduled backup mode, that mode will use a separate mechanism (a launchd agent or similar) rather than relying on model auto-invocation.

### Alternatives considered

**Allow auto-invocation (no `disable-model-invocation`).** Rejected. Side-effecting, drive-writing operations should not be triggered without explicit user intent. The risk of a spurious backup or restore during unrelated work is too high. Even with careful prompting, the model cannot perfectly predict when a backup would be appropriate.

**Gated auto-invocation (allow invocation, but require a confirmation argument).** Considered but rejected for v0.1. A confirmation gate is complex to implement correctly and places trust in the model's judgment about when to ask for confirmation. The simpler and safer default is explicit user invocation only.

## 9. Python (stdlib-preferred) for vendored helper scripts

### Context

rehydrate helper scripts run on the user's Mac at backup and restore time. They must be portable, available without additional installation steps, and easy to audit.

macOS ships Python 3 as part of the developer tools (via `xcode-select`) and it is present on virtually every developer Mac. Python has excellent stdlib support for the operations rehydrate needs: file walking, SHA-256 hashing, JSON serialization, subprocess calls, and path manipulation.

The scripts are vendored with the plugin (not installed globally), so they must work in a variety of environments without relying on packages outside the standard library for core operations.

### Decision

All rehydrate helper scripts are written in **Python 3**, targeting the version available via `python3` on macOS (currently 3.9+ on Ventura/Sonoma/Sequoia). Core operations use only the Python standard library. Optional integrations (e.g., progress bars, rich output) may use third-party packages if they are declared optional and gracefully degrade when not available.

Scripts are located in `scripts/` within the plugin directory and are invoked by the skill via `python3 scripts/<script>.py` with explicit arguments. No global installation of rehydrate scripts is required.

### Consequences

- No extra runtime installation required for core operations. `python3` is already available.
- Scripts are readable, auditable Python — not compiled binaries.
- The stdlib constraint keeps the dependency surface minimal and the scripts inspectable.
- Python is not as fast as Go or Rust for I/O-intensive operations, but the bottleneck in a backup run is disk I/O, not CPU — Python's speed is adequate.
- Scripts must be careful to handle macOS-specific paths and behaviors (e.g., `~/Library`, extended attributes, `xattr`) using stdlib or subprocess calls to macOS tools rather than assuming a POSIX-generic environment.
- Third-party packages, if used for optional features, must be documented and must degrade gracefully when not installed (a missing progress bar is acceptable; a missing SHA-256 implementation is not).

### Alternatives considered

**Bash.** Rejected. Bash is too limited for JSON manipulation (requires `jq` as an external dependency), error handling is verbose and error-prone, and complex logic (content-addressed storage, manifest building) becomes unmaintainable. Bash is acceptable for glue and entry points; it is not acceptable as the implementation language for the core scripts.

**Node.js.** Rejected. Node.js is not guaranteed to be present on a Mac that has not had it explicitly installed. Developer Macs often have Node via `nvm` or `volta`, but the version and availability are inconsistent. Adding a Node dependency for core scripts would exclude users without Node or require a version management step during plugin installation. Python's pre-installed availability is a meaningful advantage.

**Go.** Rejected. Go requires a compile step. Distributing compiled binaries in a Claude Code plugin raises inspection and trust questions. The compile step also complicates the plugin install experience — the user would need Go installed, or the plugin would need to ship pre-compiled binaries for each supported architecture (arm64, x86_64). Python source files are directly auditable by the user without tooling.

|---|---|---|

| 2 | Content-addressed `objects/` (SHA-256, 2-level fanout) | Dedup + integrity verification at the cost of non-human-readable filenames |
| 3 | Sandboxed test restore as verify | Real code path exercise without live-state risk |
| 4 | iCloud-trusted exclusion list | Smaller, safer defaults; user must opt in to capture iCloud-managed data |
